# Fabrik Dashboard – Diagnose & Heilung

**Datum:** 2026-03-01
**Ziel:** ai-brain-01 (192.168.44.10) – Fabrik Dashboard erreichbar machen
**Durchgeführt von:** Infra-Manager (Claude Code)

---

## Phase 1 – Diagnose

### Ursache

**Docker Desktop war nicht gestartet.** Der Docker Daemon Socket (`/Users/larsbruckschen/.docker/run/docker.sock`) existierte nicht. Ohne Docker Daemon konnten weder der Backend-Container (FastAPI/uvicorn) noch der Frontend-Container (nginx) laufen.

### Diagnose-Details

| Prüfpunkt | Ergebnis |
|------------|----------|
| `docker compose ps` | Fehler: Docker Daemon nicht erreichbar |
| `docker ps -a` | Fehler: Socket nicht vorhanden |
| Port 3000 | CLOSED – kein Listener |
| Port 8000 | CLOSED – kein Listener |
| `.env` Datei | Vorhanden, korrekt konfiguriert |
| `docker-compose.yml` | Gültig (backend + frontend Services) |
| `curl localhost:3000` | HTTP 000 – keine Verbindung |
| Python-Version (nativ) | 3.14 – zu neu für pydantic-core 2.27.1 (max Python 3.13) |

### Zusätzliche Erkenntnisse

- `start.sh` erkennt macOS und delegiert an `start-dev.sh` (Backend nativ, Frontend in Docker)
- **Dev-Modus nicht nutzbar**: Python 3.14 auf dem Host ist inkompatibel mit den Backend-Dependencies (pydantic-core benötigt PyO3 ≤ 3.13)
- Voller Docker-Modus funktioniert, da das Backend-Dockerfile Python 3.12-slim verwendet
- Kein Python-venv vorhanden

---

## Phase 2 – Heilung

### Durchgeführte Fixes

1. **Docker Desktop gestartet**: `open -a Docker` → Daemon innerhalb von 2 Sekunden bereit
2. **Alte Container aufgeräumt**: `docker compose down` → gestoppte Container aus vorheriger Session entfernt
3. **Container neu gebaut und gestartet**: `docker compose up -d --build`
   - Backend-Image: `fabrik_dashboard-backend` (Python 3.12-slim, uvicorn)
   - Frontend-Image: `fabrik_dashboard-frontend` (Node 20 Build → nginx:alpine)
   - Backend-Healthcheck bestanden → Frontend gestartet

### Verifikation

| `docker compose ps` | Beide Container `Up`, Backend `healthy` |
| `curl http://localhost:3000` | **HTTP 200** (Frontend HTML) |

| `curl http://localhost:3000/health` | **HTTP 200** – `{"status":"healthy","machines_tracked":10}` |
| `curl http://localhost:3000/api/machines` | **HTTP 200** – 10 Maschinen erkannt |

## Bekannte Nebeneffekte (kein Blocker)

1. **GitHub-Sync-Fehler**: `Label.description` Pydantic-Validierungsfehler wenn GitHub-Labels keine Description haben (`None` statt String). Betrifft `/api/github/pulls` und `/api/github/issues` – geben leere Arrays zurück. → Separater Bug-Fix nötig (Pydantic Model anpassen).

2. **SSH-Warnungen**: Einige VMs melden "No authentication methods available" – SSH-Key-Konfiguration innerhalb des Docker-Containers prüfen (Keys werden als Volume gemountet, ggf. Berechtigungen).

3. **Dev-Modus (macOS nativ)**: Nicht nutzbar solange nur Python 3.14 installiert ist. Fix: `brew install python@3.12` oder pydantic-core Version upgraden. Für Produktion ist der volle Docker-Modus ausreichend.

---

## Zusammenfassung

**Dashboard ist wieder erreichbar und funktionsfähig unter `http://192.168.44.10:3000`.**

Hauptursache war ein nicht gestarteter Docker Desktop. Der volle Docker-Modus (`docker compose up -d --build`) wurde statt des macOS Dev-Modus verwendet, da Python 3.14 auf dem Host inkompatibel mit den Backend-Dependencies ist.
